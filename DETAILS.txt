# To-Do Project: Complete Technical Details

## 1) Project Overview

This project is a full-stack To-Do application with:
- A Spring Boot backend exposing REST APIs for users and todos.
- A React + Vite frontend for account creation, login, and todo management.
- H2 in-memory database for development/testing.

The functional model is:
- A user can register and log in.
- A logged-in user can create, view, update, and delete only their own todos (via user-scoped API paths).

## 3) High-Level Architecture

### Backend Layering
- Controller layer: receives HTTP requests and returns HTTP responses.
- Service layer: business logic and validation decisions.
- Repository layer: JPA queries and persistence.
- DTO + Mapper layer: request/response shape separation from entities.
- Exception layer: centralized error response handling.

### Frontend Layering
- pages/: top-level screens and orchestration.
- components/: reusable UI pieces.
- api/: functions that call backend endpoints.
- hooks/: stateful business logic wrappers.
- utils/: validation helpers.
- styles/: theme and component styles.

## 6) Data Model

### User Entity
Fields:
- id (Long, PK, auto-generated)
- username (unique, required)
- email (unique, required, email format)
- password (required, min 6, write-only in JSON, stored hashed with BCrypt)
- createdAt (LocalDateTime)
- todos (OneToMany mapped by user)

### Todo Entity
Fields:
- id (Long, PK, auto-generated)
- title (required, 3-100 chars)
- description (required, max 500 chars)
- completed (Boolean)
- createdAt (LocalDateTime)
- user (ManyToOne, required FK)

Relationship:
- One User -> Many Todos


## 7) API Endpoints

Base URL:
- http://localhost:8080

### User APIs
- POST /api/users
	- Create user account.
- POST /api/users/login
	- Login using username + password; service checks the password against the stored BCrypt hash.
- GET /api/users
	- Fetch all users.
- GET /api/users/{userId}
	- Fetch user by id.
- PUT /api/users/{userId}
	- Update user fields.
- DELETE /api/users/{userId}
	- Delete user account.

### Todo APIs (User-Scoped)
- POST /api/users/{userId}/todos
	- Create todo for a user.
- GET /api/users/{userId}/todos
	- Get all todos for a user.
- GET /api/users/{userId}/todos/{todoId}
	- Get single todo.
- PUT /api/users/{userId}/todos/{todoId}
	- Update todo.
- DELETE /api/users/{userId}/todos/{todoId}
	- Delete todo.
- GET /api/users/{userId}/todos/hello
	- Utility/test endpoint.


## 8) Validation Rules

### Backend Validation (DTO/Entity)
- User create:
	- username required, 3-50
	- email required, valid format
	- password required, min 6
- User update:
	- optional fields, with format/size constraints if provided
- Login:
	- username required
	- password required
- Todo create:
	- title required, 3-100
	- description max 500
- Todo update:
	- title optional (3-100)
	- description required, max 500
	- completed optional

### Frontend Validation
File: FRONTEND/src/utils/validation.js
- username regex: [A-Za-z0-9_], length 3-30
- email regex validation
- password length >= 6

Applied in:
- UserForm (registration)
- LoginForm


## 9) Current User Workflow

### A) Registration + Login
1. User opens landing page.
2. User can create account via Create Account panel.
3. User logs in via Login panel (username + password).
4. On successful login, frontend sets activeUser in memory and opens Todos screen.

### B) Todo Management (Logged-In)
1. Frontend calls todo endpoints using activeUser.id.
2. User can create, edit, mark complete/pending, and delete todos.
3. User can logout (returns to login/register screen).
4. User can delete own account from Todos screen.


## 10) Security / Access Model (Current)

Current model is simple and lightweight:
- Login checks username, then compares the provided password against the stored BCrypt hash.
- No JWT/session token yet.
- Frontend state controls which user workspace is shown.
- Todo APIs are user-scoped by path (/api/users/{userId}/...).

Important limitation:
- Passwords are no longer stored as plain text.
- The current model still does not include token-based auth, so access control is driven by the current frontend flow and user-scoped endpoints.

## 12) Error Handling Behavior

Centralized handler:
- GlobalExceptionHandler returns ErrorResponse with:
	- timestamp
	- status
	- error
	- message
	- path (currently empty string in handler)

Mapped exceptions:
- ResourceNotFoundException -> 404
- EmptyDb -> 404
- IllegalArgumentException -> 400
- Generic Exception -> 500


## 13) Runtime Configuration

File: src/main/resources/application.yaml

Key values:
- server.port = 8080
- datasource.url = jdbc:h2:mem:tododb;MODE=MySQL;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
- hibernate.ddl-auto = create-drop
- h2 console enabled at /h2-console

Note:
- create-drop recreates schema at startup and drops on shutdown.
- Data is not persistent between runs.

## 17) Known Gaps / Current Constraints

- Passwords are not hashed.
- No token-based authentication/authorization.
- API itself does not enforce identity ownership beyond userId path usage.
- Validation exceptions are not custom-formatted separately (generic handling may return 500 in some validation cases).
- README.md currently has merge conflict markers and should be cleaned.
- H2 in-memory DB means no persistence across restarts.

## 19) End-to-End Flow Summary

1. User opens frontend.
2. User registers account (Create Account panel).
3. User logs in (Login panel).
4. Frontend receives UserResponseDTO and sets active user.
5. Todo screen loads todos for that user id.
6. User performs create/update/toggle/delete todo actions.
7. Frontend calls user-scoped endpoints and refreshes local state.
8. User logs out or deletes account.
